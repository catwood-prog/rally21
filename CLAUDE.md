## Warmth laws (Rally21-Glow-Spec.md §0, amended 6 July)

> **Misses cost something. Shame costs nothing.** Every loss has a
> rescue (a friend) or a road back (embers). No notification ever
> scolds. The penguin never guilt-trips — at most it looks hopeful.

## Product conventions

- **Every free-text input ships with the dictation affordance (Cat, 6 July, via B1).** Any free-form text box anywhere in the app — check-in's grateful/learned lines, the blueprint's "not quite" correction note, v2's short-text questions, Ask Rally's composer — gets a `VoiceMicButton` (`components/VoiceMicButton.tsx`) wired to append (never replace) into the field via the Web Speech API. Spoken answers are richer than typed ones, and corrections especially benefit. Reuse the component as-is; don't rebuild the mic affordance per-screen.
- **Practice names are verb phrases by convention, not by validation** (e.g. "Meditate 10 minutes", "Walk 20 minutes", "Write one page"), never noun phrases (not "10 Min Meditation") — this is guidance for practices seeded in code or created via the admin/seed path, nudged at creation via a verb-phrase placeholder, but nothing blocks saving a free-form custom name. Because of that, every sentence surface built from a practice name must handle any name gracefully instead of assuming a verb start: check `isVerbPhrasePractice` (`constants/strings.ts`, backed by `PRACTICE_VERB_STARTERS`) and degrade — the Today headline reads "today you {practice} with your circle" for a recognized verb start, "today: {practice}, with your circle" otherwise; the check-in headline's serif accent (`lib/practice-accent.ts`) falls back to the word "practice" for anything that doesn't match. Apply the same check to any new sentence built from a practice name.
- **Navigation calls must always use clean paths** (e.g. `router.push('/today')`), never file-system group syntax (`'/(app)/today'`). Group-qualified paths happen to typecheck and build today, but they silently break at runtime the moment a screen moves to a different group — as happened when `today.tsx` moved into `(app)/(tabs)/`. Clean paths are stable across any folder reorganization.
- **All screens must look right at both 390px and desktop widths.** The app is phone-designed (`app/+html.tsx` caps `#root` at `max-width: 480px`, centered, with the warm bg filling the sides on wider viewports), but that only fixes overall layout width — individual screens still need to be checked at both sizes, since a phone-width design can still overflow, misalign, or look sparse once centered in a wider browser window.
- **The brandmark is a shared component** (`components/Brandmark.tsx`) — "Rally" in Bricolage Grotesque extra-bold ink, immediately followed by "21" in Instrument Serif italic gold, no space between. Every screen's header renders it via `<Brandmark />` (or `<Brandmark light />` on dark backgrounds like the timer). Never recreate it with a per-screen `Text` + style — a screen with the wordmark typed out inline instead of the component is a bug.
- **No code may assume a single or primary circle.** Every circle-scoped screen takes an explicit `circleId`. There is no `getMyPrimaryCircle`-style fallback anywhere in the codebase — when a screen is reached without a `circleId` (stale link, refresh, direct nav), fetch the user's circle list and either use it unambiguously (exactly one circle) or ask which circle via a small picker (more than one). Guessing at "the first one" is how the invite-screen bug (wrong circle's code shown) happened.
- **Screens showing shared data must refetch on focus** — never assume data loaded once stays current. Use the `useFocusEffect(useCallback(() => { load(); }, [load]))` pattern (already used by Today, the circle tab, wall, weekly, journal, settings, my-practices, reflection, welcome-back, and invite) on any screen that displays circle, member, or membership data, since a tab screen stays mounted across navigations and a plain `useEffect` on mount will only ever show what was true the moment it first mounted — a rename, a join, or a leave elsewhere won't show up without a hard refresh otherwise.
- **Circle rename has no host handover.** Only the circle's original `created_by` can rename it (enforced by RLS, `created_by = auth.uid()`), even after they leave. If that matters later (creator leaves, someone else wants to rename), it needs a deliberate ownership-transfer feature — don't bolt renaming permission onto membership role in the meantime.
- **Custom practices are private by default, never silently public.** A practice's `is_shared` column controls visibility: seeded practices (`created_by is null`) are always shared; a custom practice starts private and only becomes shared the moment a public circle uses it (`create_circle`'s `p_is_public` path sets `is_shared = true` — one-directional, there's no "make private again" today). This is enforced by RLS on `practices` (`created_by is null or is_shared = true or created_by = auth.uid()`), not by client-side filtering, so every surface that lists practices (the find-a-practice grid, open-circle counts, My practices) gets the rule for free — other people's private practices are simply absent from any query result, not just hidden in the UI.
- **Leaving a circle has no host handover either.** If the creator leaves and others remain, the circle just keeps running creator-less — nothing reassigns `created_by`, so renaming (and any future creator-only action) becomes unavailable until a real ownership-transfer feature exists. Leaving never touches the leaver's completions/reflections (history belongs to the member). A circle that drops to zero members is marked `circles.is_active = false` rather than deleted — its row, history, and invite code all survive, and `join_circle_by_code`/`join_public_circle` flip it back to active the moment anyone (re)joins. `is_active = false` currently only affects public-circle discovery (`list_public_circles`, `count_open_circles_by_practice` exclude inactive circles); nothing else in the app checks it.
- **Color roles: gold = action, green = progress/alive, plum = inner life, scarce by design (decided 6 July with Cowork-Claude).** Gold is the primary CTA color (Check in, Save, invite/join actions). Green is progress and togetherness: progress bars, done/checked-in states, the active tab bar (except Journal, see below), "invite someone" and "+ add a circle" (togetherness actions keep green deliberately even though they're not progress bars). Plum (`colors.plum` / `colors.plumSoft`) is the inner-life layer — journal, reflections, the day-14 observation — and only that: the Journal tab's active-state icon/label, "Your journal" and "Something we noticed" links on Today, the day-14 observation card's border/label/serif accent and its response chips, and the reflection question's label/answer chips on check-in. Never plum on a CTA, progress bar, check-in state, non-journal nav, or an invite/join flow — scarcity is what makes it read as "your private picture" instead of just another accent color. Quiet navigation links that aren't progress-related (e.g. "This week") stay ink/muted rather than green or plum.
- **Cover-a-friend is a celebrated gift, never a debt.** `completions.kind` (`'self' | 'covered'`) and `completions.covered_by` record that another member logged someone's day for them. All four business rules — can't cover yourself, one cover per member per day, only circle members can cover, can't cover someone who already checked in today — are enforced entirely in RLS (a second INSERT policy on `completions`, OR'd with the self-completion policy), not in application code. A covered day counts toward the circle's glow (`getCirclePresence`/`computeSignal` read every completion regardless of `kind`) but is deliberately excluded from the covered member's own weekly show-up count (`getMyCompletions` filters `kind = 'self'`) and never creates a reflection — the circle stays warm, personal history stays honest. No copy anywhere (the cover screen, the covered member's next-day note, the wall entry) may read as a score, an "owed," or guilt; the covered member's own copy is always restart-framed. There is no gender data in this app, so cover-flow copy avoids pronouns. The "wave hello" alternative is a plain `wall_messages` insert with no completion — lighter touch, no signal impact.

## Code conventions (written down from established practice — audit §3c)

- **Style tokens over literals.** Any card surface spreads `...cardShadow` from `constants/theme.ts`; chips spread `chipShape`/`chipTextShape`. Colors always come from the `colors` palette (`colors.greenSoft`, `colors.errorRed`, `colors.gold`, …) — never ad-hoc hex literals in screen styles. If a color is needed twice, promote it to the palette.
- **`constants/strings.ts` owns user-facing copy.** Parameterized strings are functions, not raw templates, so call sites can't typo a token. New user-visible sentences go there, not inline in JSX.
- **The `load` idiom.** Data-fetching screens define `const load = useCallback(async () => {...}, [...])` and wire it through `useFocusEffect(useCallback(() => { load(); }, [load]))` — keep the naming, it makes the refetch-on-focus convention greppable.
- **Realtime channels use per-mount sequence-numbered topics** (e.g. `` `circle-wall-${circleId}-${++seq}` ``) because supabase-js reuses channels by topic name across mounts — reusing a topic re-triggers the "cannot add callbacks after subscribe" crash. Every new subscription follows this pattern, one subscription per screen lifecycle, always removed on cleanup.
- **Destructive-ish actions confirm inline, not in a modal** — a state boolean swaps the button for an inline confirm/cancel card (see `settings.tsx` delete-account, leave-circle). Keep this pattern for consistency.
- **Asset families live behind `@/assets/<name>/index.ts` require-maps** (e.g. `MASCOT`) — call sites never do their own relative `require()`.
- **`MyCircle` / `CircleMember` (from `lib/circle.ts`) are the canonical circle data shapes.** Don't redeclare local circle/member types on screens — extend the canonical ones if a screen needs more.
- **Module boundary:** `lib/circle-setup.ts` = practices + creating/joining circles; `lib/circle.ts` = ongoing membership (fetching, presence, rename, leave). Add new functions to the right side of that line.

## Testing

- **Running the unit tests.** `npm run test:ci` runs the full Jest suite once (`npm test` runs it in `--watchAll` mode for local dev). The runner is `jest` + the `jest-expo` preset, already configured in `package.json` — no extra setup needed. `jest.setup.js` mocks `@react-native-async-storage/async-storage` and `lib/supabase.ts` itself, since `EXPO_PUBLIC_*` env vars are inlined at build time (setting them in a Jest setup file is a no-op under `jest-expo`) and the native AsyncStorage module doesn't exist under Jest. Because of that mock, any `lib/` test exercising real Supabase calls needs to inject its dependencies (see `resolveCircleSelection`'s `deps` parameter in `lib/circle.ts` for the pattern) rather than relying on the module's real network calls.
- **Running the RPC-boundary integration tests.** `supabase/caps.integration.test.ts` and `supabase/practice-privacy.integration.test.ts` exercise real RLS/RPC enforcement — this needs a direct, privileged Postgres connection (not the anon-key REST client the app uses), since the RPCs branch on `auth.uid()`, which reads a JWT claim no anon-key call can forge. Set `SUPABASE_DB_URL` (the project's direct Postgres connection string — Supabase dashboard → Project Settings → Database → Connection string, "URI" tab, direct connection not the pooler) before running `npm run test:ci`; without it, both suites log a warning and skip. Each suite runs inside one transaction that's always rolled back in `afterAll`, so neither ever leaves rows behind regardless of what it asserts — never point `SUPABASE_DB_URL` at anything you're not comfortable temporarily inserting throwaway `auth.users`/`circles`/`memberships`/`practices` rows into. The privacy suite specifically must `set local role authenticated` before every visibility check — a direct connection's default role (`postgres`/table owner) bypasses RLS entirely regardless of the JWT claim, so skipping that role switch would silently test nothing.
- **Shared logic in `lib/` gets a test when it gains a second caller.** A helper used by exactly one screen can stay untested if it's simple enough to eyeball — but the moment a second call site starts depending on the same behavior (like `resolveCircleSelection`, shared by the circle tab, wall, and invite screens), a regression in it breaks multiple screens at once and it's cheap enough to pin down with a unit test. Add the test in the same change that introduces the second caller, not later.

## Deployment / workflow conventions

- **A feature is not "shipped" until it is live.** After every push, verify that a Vercel production deployment was created and reached READY (run the deploy status check, or trigger `npx vercel --prod` if the GitHub webhook didn't fire). Never report work as deployed based on push alone.
- **All server-side code (edge functions, migrations, cron definitions) must exist in the repo and deploy from it.** Code that only exists in a dashboard is one accident away from not existing. Edge functions live in `supabase/functions/<name>/index.ts` and deploy via `supabase functions deploy <name>`; schema/cron changes are `supabase/migrations/<timestamp>_<name>.sql`, applied via the Supabase CLI or MCP `apply_migration` and always saved back to the repo in the same change. The dashboard (or an MCP tool call) is a deployment target, never the source of truth — never leave a fix applied only remotely.

## Security conventions (security spec S1, 7 July)

- **Every new function pins `search_path = public` and grants EXECUTE explicitly — AND explicitly revokes it from `public`/`anon`.** Despite S1's original `alter default privileges in schema public revoke execute on functions from public`, a live check on 7 July (G5) found the project's actual default ACL still grants EXECUTE on new functions to `anon`/`authenticated`/`PUBLIC` automatically (confirmed via `pg_default_acl` — the revoke apparently doesn't apply at the role `apply_migration` connects as). A new function that only adds `grant execute ... to authenticated`/`service_role` and skips an explicit revoke is silently ALSO callable by anon with zero auth — for a `SECURITY DEFINER` function taking a user id as a parameter (like `get_week_for_user(p_user uuid)`), that's a real cross-user data leak, not a theoretical one. Every new function's migration must include both `revoke all on function ... from public;` and `revoke all on function ... from anon;` explicitly, immediately before the real grant — never rely on the project default alone. Both the pin and the explicit revoke+grant belong in the same migration that creates the function, never a follow-up.
- **Revoke `anon` EXECUTE unless a pre-auth surface genuinely needs it.** The default posture for every new RPC is authenticated-only; anon access is the exception, granted deliberately (and only) when a signed-out screen actually calls it (the pre-sign-in intro screens are static and call nothing). Given the finding above, this means an explicit `revoke ... from anon` line, not just omitting an `anon` grant.
- **Seed/system practices must set `is_shared = true`.** The practices SELECT policy is `is_shared = true or created_by = auth.uid()` — a system/seed practice with `is_shared` left `false` (or `created_by is null` alone) is invisible to everyone but its own row's creator check, which seed rows fail (`created_by is null`). Any seeding path (migration, admin script) must set it explicitly.
- **A SECURITY DEFINER function never accepts client-composed content destined for email or another user's surface.** If an RPC needs to notify or message someone else, it composes that copy itself from a fixed template (interpolating only trusted values like names it looked up, never a client-supplied string) — never a client-supplied subject/HTML/body param. See `send_friend_nudge`: the client passes only `circle_id`/`recipient_id`/`local_date`; the email and wall-post text are both composed server-side.
